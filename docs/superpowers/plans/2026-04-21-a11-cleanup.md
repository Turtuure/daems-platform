# A11 follow-up — drop legacy translated columns

> Dedicated cleanup PR deferred from PR 5 (content i18n). Agent A shipped a hybrid repo that reads/writes BOTH legacy columns and the new `*_i18n` tables. Dropping the legacy columns requires repo + use-case + test fixture updates that are too invasive to fold into PR 5.

**Goal:** Drop `events.title/location/description` and `projects.title/summary/description`. Make `*_i18n` tables the sole source of truth for translated content.

**Risk:** Test fixtures across 7 files seed events/projects rows via raw SQL with legacy columns. Each fixture needs updating to (1) omit legacy columns from the INSERT, (2) add a companion INSERT to `events_i18n` / `projects_i18n` so reads return meaningful title/description.

## Completed exploration (this session)

A first pass was attempted and reverted. State after revert: identical to dev (no code changes landed). The investigation produced these artefacts:

- Migration 054 SQL (draft — not committed): dropped `events.title/location/description` + `projects.title/summary/description`
- Repo refactors (draft — reverted): `SqlEventRepository` / `SqlProjectRepository` `save()` stopped writing legacy cols; `hydrate()` derived convenience scalars from `TranslationMap::rowFor(UI_DEFAULT)` with first-available fallback; `loadTranslationMap()` removed legacy-row synthesis; `updateForTenant` allowed list shrunk to chrome-only; `findRegistrationsByUserId` + `listRecentCommentsForTenant` switched to `COALESCE((SELECT title FROM events_i18n ...), ...)` subqueries; `listForTenant` / `listAllStatusesForTenant` search switched to `EXISTS (SELECT 1 FROM projects_i18n ...)`
- Use-case split (draft — reverted): `UpdateEvent` and `AdminUpdateProject` routed `title/location/description`/`title/summary/description` fields through `saveTranslation(SupportedLocale::uiDefault(), ...)` (with merge-from-existing to preserve unchanged fields) while `chromeFields` went to `updateForTenant` as before
- In-memory fakes (draft — reverted): `InMemoryEventRepository::saveTranslation` + `InMemoryProjectRepository::saveTranslation` rebuilt the stored entity with the new locale's values merged into a fresh `TranslationMap`, matching SQL-repo semantics

All four PHPUnit test suites stayed green on draft Unit (512) + E2E (89) + PHPStan lvl 9 (0 errors). Integration failed with 24 errors + 1 failure — all in test fixtures that seed events/projects rows through raw PDO INSERT statements including translated columns. That is the scope that remains.

## Plan (numbered tasks)

### Task 1: Create migration 054
- Create `database/migrations/054_drop_translated_columns_from_events_projects.sql`
- Apply to dev DB via `php scripts/apply_pending_migrations.php`
- Commit

### Task 2: Refactor `SqlEventRepository`
- `save()` — remove `title`, `location`, `description` from INSERT column list + ON DUPLICATE KEY UPDATE
- `hydrate()` — stop reading `self::str($row, 'title')` etc; derive convenience scalars from `$translations->rowFor(SupportedLocale::uiDefault())` with first-available fallback
- `loadTranslationMap()` — drop legacy-scalar parameters + the legacy-row synthesis branch
- `updateForTenant()` — drop `title`, `location`, `description` from `$allowed` list
- `findRegistrationsByUserId()` — replace `SELECT e.title` with a `COALESCE` subquery on `events_i18n`

### Task 3: Refactor `SqlProjectRepository`
- Mirror Task 2 pattern
- `listForTenant` + `listAllStatusesForTenant` search — replace `title LIKE ? OR summary LIKE ?` on parent table with `EXISTS (SELECT 1 FROM projects_i18n ...)` subquery
- `listRecentCommentsForTenant` — replace `SELECT p.title` with `COALESCE` subquery on `projects_i18n`

### Task 4: Refactor use cases
- `src/Application/Backstage/UpdateEvent/UpdateEvent.php` — split `$chromeFields` vs `$translationFields`; translated-field path merges with existing fi_FI row and calls `saveTranslation(SupportedLocale::uiDefault(), $merged)`
- `src/Application/Backstage/AdminUpdateProject/AdminUpdateProject.php` — same pattern

### Task 5: Update in-memory fakes
- `tests/Support/Fake/InMemoryEventRepository.php::saveTranslation` — rebuild stored `Event` from new translation row so subsequent `findByIdForTenant()` reflects the write (match SQL repo semantics)
- `tests/Support/Fake/InMemoryProjectRepository.php::saveTranslation` — same

### Task 6: Update migration-level markers
- `tests/Integration/Infrastructure/SqlEventRepositoryI18nTest.php` — bump to `runMigrationsUpTo(56)`
- `tests/Integration/Infrastructure/SqlProjectRepositoryI18nTest.php` — same

### Task 7: Fix 7 test fixtures (largest scope)
For each file listed above, every raw INSERT into `events` or `projects`:
- Remove `title`, `location`, `description` / `title`, `summary`, `description` from the column list
- Immediately after, INSERT a companion `events_i18n` / `projects_i18n` row (locale = `fi_FI`) carrying those values
- Update any `SELECT ... FROM events` / `FROM projects` assertions that read translated columns to either JOIN `*_i18n` or fetch via `repo->findByIdForTenant()`

Pattern for events fixtures:
```sql
-- before:
INSERT INTO events (id, tenant_id, slug, title, type, event_date, location, is_online, description, status)
VALUES (?, ?, ?, ?, ?, ?, ?, ?, ?, ?)

-- after:
INSERT INTO events (id, tenant_id, slug, type, event_date, is_online, status)
VALUES (?, ?, ?, ?, ?, ?, ?)
INSERT INTO events_i18n (event_id, locale, title, location, description)
VALUES (?, 'fi_FI', ?, ?, ?)
```

Pattern for projects fixtures is analogous with `title`, `summary`, `description`.
